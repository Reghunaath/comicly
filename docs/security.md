# Security Policy — Comicly

## OWASP Top 10 Checklist

| # | Vulnerability | Comicly Mitigation | Status |
|---|--------------|-------------------|--------|
| A01 | Broken Access Control | Supabase Auth for identity. Ownership checks in `delete-comic` and `get-library` handlers. Guest vs authenticated rules enforced in application code via `getOptionalUser` / `getRequiredUser`. | ✅ |
| A02 | Cryptographic Failures | No custom crypto. Supabase handles password hashing (bcrypt) and session tokens (JWT). HTTPS enforced by Vercel on all traffic. No plaintext secrets stored anywhere. | ✅ |
| A03 | Injection | All database access goes through the Supabase client (parameterised queries — no raw SQL). AI prompts use template string interpolation with user input HTML-escaped at display. Zod schemas validate all API inputs before they reach the DB or AI pipeline. | ✅ |
| A04 | Insecure Design | Auth is optional by design for comic creation but required for data persistence (library, delete). Shared comics are intentionally public and read-only. Generation mode is server-enforced, not client-controlled. | ✅ |
| A05 | Security Misconfiguration | `SUPABASE_SERVICE_ROLE_KEY` and `GEMINI_API_KEY` are server-side only — never prefixed with `NEXT_PUBLIC_`. Supabase RLS is enabled. Storage bucket is public (intentional — all images are user-created content intended for sharing). No default credentials in use. | ✅ |
| A06 | Vulnerable Components | `npm audit --audit-level=high` runs in every CI pipeline and blocks merges on high/critical findings. | ✅ |
| A07 | Auth Failures | Supabase Auth handles brute-force protection, session management, and token refresh via `middleware.ts`. No custom auth logic. Session cookies are `httpOnly` and `Secure`. | ✅ |
| A08 | Data Integrity Failures | Script JSON from Gemini is parsed inside `try/catch`, structurally validated via `validateScript()`, and Zod-validated at the API boundary. User-submitted script edits are re-validated before saving. Image URLs are system-generated (Storage public URLs), never user-provided. | ✅ |
| A09 | Logging Failures | Vercel function logs capture errors. Supabase dashboard provides auth and database logs. Error responses return `{ error: "Human-readable message" }` — no stack traces or internal details leaked to clients. No sensitive data (tokens, keys, PII) in logs. | ✅ |
| A10 | SSRF | Gemini API calls use hardcoded `googleapis.com` endpoints (from the official SDK). No user-controlled URLs are fetched server-side. Character sheet and page image URLs are generated by Supabase Storage, not user-provided. | ✅ |

## Secret Handling

| Secret | Storage | Scope |
|--------|---------|-------|
| `GEMINI_API_KEY` | Vercel env vars | Server-side only |
| `SUPABASE_SERVICE_ROLE_KEY` | Vercel env vars | Server-side only — never `NEXT_PUBLIC_` |
| `NEXT_PUBLIC_SUPABASE_URL` | Vercel env vars | Public (safe) |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Vercel env vars | Public (safe, RLS-protected) |
| `CLAUDE_CODE_OAUTH_TOKEN` | GitHub Secrets | CI only |
| `GITHUB_TOKEN` | GitHub Secrets (auto) | CI only |

**Rule:** Any variable without `NEXT_PUBLIC_` is automatically server-only in Next.js. Never add a secret to a `NEXT_PUBLIC_` variable.

## Gitleaks — Secrets Detection

Gitleaks runs automatically in the `security` CI job on every push and PR. To run it locally before committing:

```bash
# Install (one-time)
brew install gitleaks          # macOS
# or download from https://github.com/gitleaks/gitleaks/releases

# Scan the repo
gitleaks detect --source . --verbose

# Scan staged changes only (pre-commit style)
gitleaks protect --staged --verbose
```

To add a local pre-commit hook:

```bash
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/sh
gitleaks protect --staged --verbose
EOF
chmod +x .git/hooks/pre-commit
```

## API Error Response Convention

All API routes return errors in this shape — no stack traces, no internal details:

```json
{ "error": "Human-readable message" }
```

Status codes used:
- `400` — validation error or invalid state transition
- `401` — not authenticated
- `403` — authenticated but not authorised (wrong owner)
- `404` — resource not found
- `500` — unexpected server error (message is always "Internal server error")
